Explaining Medical MIL
Inspired by: Ilse et al., Attention-based 
Deep Multiple Instance Learning  (ICML 2018)
1. Instance Feature Matrix (Input)
Bag: Contains K cell instances.
Encoder (CNN): Maps each cell to a 500-D vector.
Input Matrix H: [K, 500].

2. Gated Attention Network
Assessment (tanh branch): H [K, 500] * V [500, 128] -> tanh = [K, 128].
Gating (sigmoid branch): H [K, 500] * U [500, 128] -> sigmoid = [K, 128].
Final Scoring: Multiply combined branches by W_transpose [128, 1] → Softmax.
Output: Attention weights A [K, 1].

3. Aggregation
Inputs: Weights A [K, 1] and Original features H [K, 500].
Dot Product: Transpose A into [1, K] * H [K, 500].
Output: Bag representation vector z [1, 500].

4. Bag-Level Prediction
Classifier: Vector z [1, 500] * Weights [500, 1].
Calculation: [1, 500] * [500, 1] = [1, 1] (A single scalar).
Output (sigmoid): Squeezes the scalar into a range between 0 and 1.
bag = torch.Size([1, 10, 1, 28, 28])

#                       batch K channel H  W

Dataset Preparation: Source Data BERT4ETH
Feature Vector Construction (68-dim)

Each transaction window is represented by 68-dimensional feature vector:
[64-dim] BERT4ETH Embedding.
[4-dim] Hand-crafted Heuristics:
+ z_amount: Z-score of the transaction value relative to that account's own history.
+ density: Number of transactions within ±30 min window (burst indicator).
+ value_ratio: Cumulative IN / (cumulative OUT + ε)= victim funds accumulating.
+ counterparty_novelty: 1 if the counterparty is a first-time address
Normalization is per-account (Z-score across that account's own transactions), not global.

Sliding Window Formulation

Why Sliding Windows?
A phishing account may have up to 60,410 transactions, the laundering burst (typically 2–5 transactions) is overwhelmed by noise.
Window Parameters
· Window size W = 200 transactions
· Stride S = 50 transactions
· Coverage guarantee: Each transaction appears in at most floor(W/S) = 4 windows
· Total windows across dataset: 40,797

Why Build a Transaction-Level Ground Truth?
- Problem: Existing labels (e.g., phisher_account.txt) are Account-Level. They tell us WHO the criminal is, but not WHEN the crime happened.
- Solution: We must build a Transaction-Level Ground Truth to evaluate Forensic Localization (the model's ability to pinpoint the exact time of the crime).
The Definition of a Phishing Burst
1. The Trap (IN): Normal accounts sending ETH INTO the phishing account.
2. The Laundering (OUT): The phisher sending ETH OUT to Tornado Cash or via massive bulk transfers.
Phishing Burst Transaction Flow
LOOP
True Data Sources
Base Accounts: The 7,067 phishing accounts and 28,272 normal accounts are loaded exactly from the BERT4ETH/Data/ directory
The Algorithm Pipeline
1. Data Ingestion
Load anchor sets into RAM: phishers, normal_accounts, tornado_endpoints.
2. Victim Filtering
Use Pandas isin(): Keep transactions where Sender is Normal AND Receiver is Phisher.
3. Merging & Sorting
Merge IN and OUT transactions per phisher.
Use sort_values(): Order chronologically by Unix Timestamp.
4. Temporal Clustering
Calculate time gaps between victim transactions.
Split into new clusters if gap > 30 days.
Use max(key=len): Pick the densest cluster. Mark start_tx_idx.
5. Endpoint Detection
Slice matrix: Extract OUT-transactions within a strict 72-hour limit.
Tornado Match: Use isin() to check if To-Address matches Tornado Cash.
Fallback: Use idxmax() to find the highest ETH transfer. Mark as end_tx_idx.
6. JSON Output
Save the final [start_tx_idx, end_tx_idx] boundaries for 2,483 valid accounts into the time_aware_ground_truth.json file.
